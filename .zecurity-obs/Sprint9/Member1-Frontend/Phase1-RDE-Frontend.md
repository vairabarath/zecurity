# M1 Phase 1 — RDE Frontend (Access Log Viewer + Device Management)

---

## What You're Building

Two new pages in the Admin UI:

1. **Access Log Viewer** — real-time table of RDE tunnel events showing who connected to what resource, when, and whether access was allowed or denied. Sourced from `connector_log` events emitted by `device_tunnel.rs`.
2. **Device Management** — per-workspace list of enrolled client devices with the ability to revoke a device cert.

---

## Dependency on M2

M2 adds the following to the GraphQL schema before you can run codegen:

```graphql
type ConnectorLog {
  id:           ID!
  workspaceId:  ID!
  connectorId:  String!
  message:      String!
  createdAt:    String!
}

type ClientDevice {
  id:         ID!
  userId:     ID!
  spiffeId:   String!
  commonName: String!
  createdAt:  String!
  revokedAt:  String
}

extend type Query {
  connectorLogs(limit: Int): [ConnectorLog!]!
  clientDevices:             [ClientDevice!]!
}

extend type Mutation {
  revokeDevice(deviceId: ID!): Boolean!
}
```

After M2 commits schema + resolver stubs, run:
```bash
cd admin && npm run codegen
```

You can scaffold component structure before codegen lands — just mock the data shape.

## Files to Create / Modify

### 1. `admin/src/pages/AccessLog.tsx` (NEW)

Access log table page.

```tsx
// Route: /access-log
// Sidebar link: "Access Log" — visible to ADMIN only

// Columns: Time | Connector | Message (destination:port, path=direct/shield_relay, allowed/denied)
// Auto-refresh every 10s via Apollo polling: pollInterval={10_000}
// Empty state: "No tunnel events yet."
// Loading skeleton: 5 placeholder rows
```

Key behaviour:
- Parse the `message` field from `connector_log` to extract: destination, port, protocol, path (`direct` or `shield_relay`), decision (`allowed` / `access denied`)
- Color-code: green for allowed, red for denied
- Show connector ID as a short badge (first 8 chars)
- Limit: fetch last 100 entries, newest first

Apollo query:
```graphql
query GetConnectorLogs($limit: Int) {
  connectorLogs(limit: $limit) {
    id
    connectorId
    message
    createdAt
  }
}
```

### 2. `admin/src/pages/DeviceManagement.tsx` (NEW)

Client device list page.

```tsx
// Route: /devices
// Sidebar link: "Devices" — visible to ADMIN only

// Columns: Device ID (short) | SPIFFE ID | User | Enrolled | Status (Active / Revoked)
// Revoke button: calls revokeDevice mutation, confirms with modal, refreshes list
// Empty state: "No enrolled devices."
```

Apollo query + mutation:
```graphql
query GetClientDevices {
  clientDevices {
    id
    userId
    spiffeId
    commonName
    createdAt
    revokedAt
  }
}

mutation RevokeDevice($deviceId: ID!) {
  revokeDevice(deviceId: $deviceId)
}
```

After revoke: refetch `clientDevices` query. Show toast "Device revoked."

## Notes

1. `revokeDevice` on the Controller side writes `revoked_at` on the `client_devices` row. The CRL is regenerated by the Controller and the Connector picks it up on its 5-min refresh cycle. The client will be rejected on their next tunnel attempt.
2. Do not add a real-time subscription — polling every 10s is sufficient for the demo.
3. Device IDs in the access log come from the SPIFFE ID embedded in the client cert — no extra lookup needed.
